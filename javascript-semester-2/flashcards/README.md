# Проект Flashcards — Описание Архитектуры и Реализации

Данный проект представляет собой веб-приложение для заучивания иностранных слов (флэшкарточек), реализованное на чистом
JavaScript (Vanilla JS). Архитектура п��иложения демонстрирует глубокое понимание паттернов проектирования,
инкапсуляции, ES6+ модулей, а также управления пользовательскими интерфейсами и сложными состояниями.

## 🏗 Архитектура и основные концепции

Проект "Flashcards" структурирован как полноформатное Модульное Приложение (Modular Application), где каждый тип логики
вынесен в отдельные папки (`app/`, `models/`, `study/`, `ui/`). Это обеспечивает абсолютную масштабируемость и легкую
поддержку кода, следующего концепциям SOLID / Separation of Concerns.

### 1. Модульная структура файлов (ES6 Modules)

- Файлы разбиты на четкие слои (Models, UI, App, StudySession).
- В качестве связующего звена используется `index.js/main.js`, импортирующий главный класс `App.js`.
- Изоляция скоуп-переменных, избегание глобальных переменных в браузере (через `import/export`).

### 2. ООП и инкапсуляция состояний (Классы из `models/`)

Приложение оперирует абстракциями `Deck` (Колода) и `Card` (Карточка), которые управляют своими данными методами.
Весь корневой стейт приложения хранится внутри класса `App` в приватных свойствах `#decks`, `#activeDeckId`, `#session`
и `#editingCardId`.
Только класс `App` имеет возможность манипулировать этим состоянием через свои публичные методы:

- `createDeck`, `deleteDeck`, `setActiveDeck`
- `addCard`, `updateCard`, `deleteCard` — эти методы всегда сопровождаются обязательным перестроением UI (вызов
  `render()`) и синхронизацией хранилища (`save()`).

### 3. Слой абстракции сессий (Паттерн Стратегия в `StudySession.js`)

Инициализация сессии заучивания вынесена в отдельную логику `StudySession`.

- Класс управляет активным индексом карточки (`getCurrentIndex`), статусом её разворота (`isFlipped()`), а также
  операциями навигации (`next()`, `prev()`).
- Разделение на режимы (modes — например, "Изученные" / "Неизученные"), а также перемешивание карточек (`shuffle`)
  инкапсулировано внутри этого класса, снимая нагрузку с `Deck` и `App`. Сброс или изменение сессии влечет за собой
  пересчет и подготовку.

### 4. Отделение бизнес-логики и UI (`ui/render.js` и `setupEventListeners`)

Приложение использует декларативный способ рендера элементов интерфейса: класс `App` **не генерирует напрямую HTML
внутри себя**.

- Вызовы передаются чистым функциям (Pure Functional Components) из модуля `ui/render.js` (например, `renderCard`,
  `renderPosition`, `renderListCards`, `renderDeckList`).
- Эти функции работают только с текущим состоянием (State), которые им передает `App.render()`.
- Класс `App` только биндит (св��зывает) обработчики событий (`setupEventListeners`) на элементы, передавая контекст
  через Arrow Functions (`() => this.flipCard()`), что решает проблему потери `this`.
- Каждое изменение в данных приводит к полному или частичному реактивному обновлению UI.

### 5. Персистентность данных (Local Storage API)

Состояние колод (Decks) сохраняется при любом добавлении карточки или удалении колоды.

- Класс `Deck.fromJSON(d)` умеет мгновенно восстанавливать цепочку прототипов и методов из "голых объектов данных" (
  JSON/LocalStorage), что является продвинутой практикой работы с данными.
- При загрузке (`setupState`) умная логика проверяет наличие данных, а если их нет (или всё было удалено), она
  автоматически генерирует новую дефолтную колоду (`new Deck()`), защищая приложение от фатальных ошибок пустого
  состояния.

## 🚀 Особенности реализации

- **Интерактивность**: Присутствует возможность редактирования карточек (Edit Mode: `#editingCardId`). При нажатии на
  кнопку Save система обновляет, а не добавляет карточку, если переменная не `null`.
- **Callback Functions**: Модули отрисовки принимают колбеки (`onDelete`, `onEdit`, `onSelect`), инвертируя управление
  событиями (Inversion of Control) обратно в класс `App`.
- Полная адаптивность работы с несколькими самостоятельными колодами, где сессия `StudySession` динамически
  привязывается к текущей `activeDeckId`.

Данная реализация показывает уровень сеньорного (Senior) подхода к построению Vanilla JS-приложений, мимикрирующего под
реактивные фреймворки (подобно React/Vue-подходу "one-way data flow" + "re-render"), но реализованного вручную.
