# Проект Game12 — Описание Архитектуры и Реализации

Данный проект представляет собой логическую пошаговую игру в кости "Game12", реализованную на чистом JavaScript (Vanilla JS). Код написан с соблюдением принципов объектно-ориентированного программирования (ООП), строгого разделения логики и представления, а также управления сложным игровым состоянием.

## 🏗 Архитектура и основные концепции

Архитектура построена вокруг основных игровых сущностей, каждая из которых инкапсулирована в свой собственный ES6+ класс ( `Dice`, `Player`, `Game` ). Это позволяет легко масштабировать игру, изменять количество кубиков, граней или число игроков, не затрагивая базовую логику.

### 1. Объектно-ориентированная модель (Model)
- **Класс `Dice`**: Инкапсулирует логику броска кубиков. Определяет количество кубиков и их граней, генерирует случайные значения и автоматически высчитывает сумму текущего броска.
- **Класс `Player`**: Отвечает за состояние конкретного игрока. Хранит его имя и текущий набор карточек. Обладает методами-помощниками для управления картами (`hasCard()`, `removeCard()`, `removeCards()`, `hasWon()`).
- **Класс `Game`**: Выступает в роли главного контроллера (Controller). Управляет очередностью ходов (`switchPlayer`), валидацией действий (`getAvailableMoves`, `playMove`, `skipMove`) и общим состоянием игры (`awaitingMove`, `gameOver`).

### 2. Управление состоянием (Константы STATUS)
Вместо магических строк и разрозненных флагов в функции `playMove()`, возвращаемые результаты строго типизированы через объект-константу `STATUS`:
- `CONTINUE`, `WIN`, `INVALID`, `INVALID_SKIP`, `GAME_OVER`, `MUST_MOVE_FIRST`.
Такой подход делает обработку игровых событий предсказуемой и надежной (в функции `handleMoveResult`), исключая ошибки при непредсказуемых сценариях.

### 3. Разделение бизнес-логики и UI (Separation of Concerns)
В проекте четко проведена граница между состоянием игры (классы) и манипуляциями с DOM.
- Логика классов **ничего не знает о браузере** (нет `document.querySelector` внутри классов).
- Функции рендера (`renderDice`, `renderPlayers`, `renderMoveButtons`) работают исключительно как слой представления (View), который рисует UI на основе текущего состояния объекта `Game`.
- Это классический паттерн проектирования, который сильно упрощает поддержку и потенциальное покрытие кода Unit-тестами.

### 4. Динамическая генерация интерфейса (DOM Manipulation)
Кнопки действий (убрать по отдельности, убрать сумму, пропустить ход) генерируются в DOM абсолютно динамично в функции `renderMoveButtons(moves)`. 
Их рендер зависит от результатов метода `getAvailableMoves()`, который анализирует массив уникальных значений кубиков и сопоставляет их с оставшимися карточками текущего игрока, реализуя сложную бизнес-логику игры в элегантном математическом виде (`Set`, `filter`, `includes`).

## 🚀 Особенности реализации
- **Масштабируемость**: Инициализация игры `new Game(2, 6, 3)` (2 кубика по 6 граней, 3 игрока) наглядно показывает, что конфигурация игры гибкая благодаря параметризации конструкторов классов.
- **Обработка массивов**: Активное использование современных методов массивов (`reduce`, `filter`, `forEach`, `includes`) и Spread-операторов (`...new Set()`) для работы с комбинациями кубиков и карт.
- Защита от ошибочных действий пользователя через валидацию текущего стейта (`awaitingMove`, `gameOver`).

Данная реализация показывает качественное понимание объектной декомпозиции, принципов DRY (Don't Repeat Yourself) и грамотной связки JavaScript с DOM-окружением.
